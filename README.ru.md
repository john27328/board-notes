# Board Notes

Интерактивные канбан-доски для Obsidian, которые рендерятся из одного code-блока и полностью опираются на frontmatter заметок.

Никакого файла-кеша. Никакой отдельной базы данных. Порядок колонок и карточек хранится прямо в самих заметках — всё остаётся текстовым, диффится в git и нормально мержится.

*(English version: [README.md](README.md))*

## Зачем

Такие плагины, как [obsidian-projects](https://github.com/marcusolsson/obsidian-projects), хранят состояние доски (порядок карточек, конфиг вида) в файле-кеше `data.json` внутри папки плагина. Этот кеш может разойтись с реальным содержимым заметок, ломается из-за проблем с кодировкой и плохо мержится между машинами и в git.

Board Notes устроен наоборот: каждая доска описывается одним ` ```board ` code-блоком, вставленным в заметку. Плагин читает карточки прямо из `metadataCache` по тегу и пишет статус и порядок карточки обратно в её же frontmatter. Вне файлов vault ничего не кешируется.

## Возможности

- **Канбан-доска** из code-блока — drag-and-drop между колонками и внутри колонки
- **Поиск** по названию, описанию, рекомендации и тегам
- **Фильтры** по тегам, по любому списковому полю frontmatter (жанры, метки — что угодно из конфига) и по видимости колонок
- **Контролируемый словарь** — задаёшь список допустимых значений для поля (например, жанры, теги), чтобы редактор выбирал из фиксированного списка, а не печатал текст вручную и не плодил почти-дубликаты
- **Карточка внутри заметки** (` ```card `) — рендерит оценку, ссылку, описание и рекомендацию прямо в теле заметки
- **Редактор тегов внутри заметки** (` ```tags `) — позволяет редактировать поля, управляемые словарём, прямо из самой заметки, а не только с доски
- Команда и пункт контекстного меню файла для редактирования словарных полей активной заметки даже без открытой доски
- Быстрое создание заметок из шаблона, с уже проставленным статусом нужной колонки
- **Статус прямо из карточки** (` ```card `) — ряд чипов с колонками доски, клик сразу меняет `statusField` в frontmatter, без похода на доску
- **Настройки доски** (кнопка ⚙ в тулбаре) — папка, шаблон, колонки, значения словаря и вид карточки редактируются в модалке; переименование колонки или значения словаря массово обновляет все карточки, которые его использовали
- **Команда «Создать новую доску»** — мастер (тег, папка, шаблон, колонки), который генерирует готовый ` ```board ` code-блок в новой заметке
- **Централизованный вид карточки** — поля/ссылки/подписи для ` ```card ` можно задать один раз в конфиге доски (`card:`), а не копировать в каждый шаблон и каждую заметку; поддерживает несколько ссылок одновременно (не только одну)
- **Автоматические даты** — при создании заметки заполняет пустое `created` сегодняшней датой, а при каждом изменении обновляет `updated` (формат `ГГГГ-ММ-ДД`)

## Установка

### Вручную

1. Скачай `main.js`, `manifest.json` и `styles.css` из [последнего релиза](../../releases/latest) (или собери сам, см. ниже).
2. Создай папку `<твой-vault>/.obsidian/plugins/board-notes/`.
3. Положи туда эти три файла.
4. В Obsidian: **Settings → Community plugins**, обнови список плагинов, включи **Board Notes**.

### Через BRAT (авто-обновляемая бета-установка)

1. Установи community-плагин [BRAT](https://github.com/TfTHacker/obsidian42-brat).
2. В настройках BRAT — **Add Beta Plugin**, вставь URL этого репозитория.
3. После этого включи **Board Notes** в Community plugins.

### Каталог community-плагинов

Пока не отправлен в каталог. Когда будет принят — установка будет доступна прямо из **Settings → Community plugins → Browse**.

## Сборка из исходников

Нужен Node.js (проверено на v20+).

```bash
git clone <url-этого-репозитория>
cd board-notes
npm install
npm run build       # разовая продакшн-сборка -> main.js
npm run dev          # watch-режим, пересобирает при сохранении
```

Если репозиторий подключён как `plugins/board-notes` внутри сопутствующего репозитория
хранилища, команда `npm run deploy` проверяет TypeScript, создаёт production-сборку и
копирует `main.js`, `manifest.json` и `styles.css` в
`notes/.obsidian/plugins/board-notes`.

`npm run build` даёт минифицированный `main.js` без source map. `npm run dev` даёт неминифицированную сборку с inline source map и следит за изменениями — направь его на скопированную/симлинкнутую папку плагина внутри реального vault для разработки вживую.

## Быстрый старт

Добавь code-блок в любую заметку:

````markdown
```board
tag: "#book"
folder: Книги
columns:
  - хочу прочитать
  - читаю
  - прочитано
```
````

Любая заметка с тегом `#book` становится карточкой, сгруппированной по колонкам через поле `Статус`/`Status` в её frontmatter (имя поля настраивается). Перетащи карточку в другую колонку — статус изменится; перетащи внутри колонки — изменится порядок. Оба действия сразу пишутся обратно в frontmatter карточки.

## Справочник по конфигурации

Все опции читаются из YAML внутри ` ```board ` блока.

| Ключ | Тип | По умолчанию | Описание |
|---|---|---|---|
| `tag` | строка | — (обязателен) | Тег, определяющий какие заметки становятся карточками этой доски, например `"#book"`. |
| `folder` | строка | корень vault | Папка, в которой создаются новые карточки по кнопке «+ добавить». |
| `template` | строка | — | Путь к заметке-шаблону. Новые карточки создаются из неё, с уже подставленным статусом выбранной колонки. |
| `statusField` | строка | `Статус` | Поле frontmatter, по которому карточки группируются в колонки. |
| `orderField` | строка | `Порядок` | Поле frontmatter, в котором хранится позиция карточки внутри колонки. Выбирай имя, которое не конфликтует с уже существующим полем. |
| `nameField` | строка | — | Поле frontmatter для заголовка карточки. Если не задано — берётся поле `Название`, иначе имя файла. |
| `columns` | string[] | вычисляется | Явный, упорядоченный список значений статуса для колонок. Если не задан — колонки выводятся из фактически встречающихся значений. |
| `exclude` | string[] | `[]` | Пути (относительно vault), которые нужно исключить из доски, даже если они помечены тегом (например сам файл шаблона, если у него тот же тег). Заметка, которая содержит сам блок доски, исключается автоматически всегда. |
| `facets` | string[] | `[]` | Поля frontmatter, которые нужно вывести как строки фильтров-чипов в тулбаре (в дополнение к автоматическому ряду фильтра по тегам, который показывается всегда, если у карточек есть дополнительные теги). |
| `vocab` | map строка → string[] | `{}` | Контролируемый словарь по полю. Любое поле, перечисленное здесь, получает редактируемую панель чипов (в блоке ` ```tags `, в команде/модалке словаря и прямо на карточке), ограниченную этими значениями. |
| `single` | string[] | `[]` | Подмножество полей из `vocab`, которые хранят одиночное скалярное значение (не список) — например поле приоритета или типа. Редактирование таких полей заменяет значение, а не переключает принадлежность массиву. |
| `meta` | string[] | `Год`/`Год выпуска` + `Оценка` | Поля frontmatter, показываемые на лицевой стороне карточки в виде доски. По умолчанию год + оценка (для обратной совместимости) — задай явно для досок без этих полей. |
| `showTags` | boolean | `true` | Поставь `false`, чтобы скрыть автоматический ряд фильтра по тегам. Полезно, если заметки содержат случайные настоящие теги Obsidian, не связанные с доской (например, буквальный `#include` в код-сниппете индексируется как тег и засоряет фильтр). |
| `flat` | boolean | `false` | Полностью отключает Kanban-колонки, рендерит все подходящие карточки одним фильтруемым гридом. Для справочных индексов (FAQ, глоссарии) с тематическими тегами, но без рабочего статуса — `statusField`/`orderField`/`columns` при этом игнорируются. |
| `autoArchive` | объект | — | Автоматически переносит карточки из `source` в `target` через `afterDays` дней после последнего изменения статуса. Поле даты — `statusChangedField`, по умолчанию `Статус изменён`. Проверка запускается при старте Obsidian и затем каждый час. |
| `card` | объект | `{}` | Централизованные настройки для блока ` ```card ` (см. ниже) — `fields`, `links`, `labels`, `ratingField`, `recField`. Применяются к любой заметке с тегом этой доски, у которой свой ` ```card ` блок пустой. |

### Блок `card`

Рендерит компактную выжимку frontmatter заметки — предполагается, что он живёт внутри самой заметки (например, в её шаблоне) как читаемая альтернатива голой панели Properties.

Каждое поле можно редактировать по клику — пустые показывают плейсхолдер «+ имя поля», заполненные — своё значение; клик по любому из них превращает поле в инпут (или textarea для длинных полей вроде описания/рекомендации), сохранение по blur или Enter, Escape отменяет. Это основной способ заполнить или поправить поле после того, как ты скрыл панель Properties.

````markdown
```card
fields:
  - Оценка
  - Кинопоиск
  - Описание
  - Рекомендация
ratingField: Оценка
links:
  - field: Кинопоиск
    label: "Открыть на Кинопоиске ↗"
recField: Рекомендация
labels:
  Id: "ID"
```
````

| Ключ | По умолчанию | Описание |
|---|---|---|
| `fields` | `[Оценка, Кинопоиск, Описание, Рекомендация]` (или из `card` доски) | Какие поля frontmatter показывать и в каком порядке. Пустые/отсутствующие поля молча пропускаются. |
| `ratingField` | `Оценка` | Рендерится как `★ <значение>`. |
| `links` | `[{field: Кинопоиск}]` | Список ссылок — каждая рендерится своей строкой с кликабельной ссылкой (если значение похоже на URL) и своим карандашом-редактором. Можно указать несколько, например Pyrus + отдельная ссылка на merge request. |
| `linkField` / `linkLabel` | — | Старый способ задать **одну** ссылку — эквивалентно `links: [{field: linkField, label: linkLabel}]`. Продолжает работать, `links` и `linkField` не смешивай в одном блоке. |
| `recField` | `Рекомендация` | Рендерится курсивом с акцентной полосой слева. |
| `labels` | `{}` | Карта имя-поля → подпись для любого другого поля из `fields`. Рендерится как небольшая строка «Подпись: значение» вместо целого абзаца — используй для короткой метаинформации (ID, счётчики), а не для прозы. Поля из `fields` без подписи и не совпадающие ни с одной из ролей выше рендерятся обычным абзацем (годится для длинного текста вроде описания). |
| `showStatus` | `true` | Поставь `false`, чтобы скрыть ряд чипов со статусом (см. ниже). |

Если ни у одного поля из `fields` нет значения, блок покажет небольшую заглушку «Поля не заполнены» вместо пустоты.

#### Централизованная конфигурация

Если сам ` ```card ` блок в заметке **пустой** (без `fields`), плагин ищет доску, чей `tag` совпадает с тегом заметки, и берёт `fields`/`links`/`labels`/`ratingField`/`recField` из её конфига (ключ `card:` в блоке ` ```board `):

````markdown
```board
tag: "#book"
folder: Книги
card:
  fields:
    - Оценка
    - Кинопоиск
    - Описание
    - Рекомендация
  links:
    - field: Кинопоиск
      label: "Открыть на Кинопоиске ↗"
```
````

Так шаблон и все заметки этого типа хранят всего один пустой ` ```card ``` `, а сам список полей/ссылок правится в одном месте — в конфиге доски (руками или через кнопку ⚙, см. ниже). Если в конкретной заметке всё же нужен свой набор полей — просто пропиши `fields`/`links` прямо в её ` ```card ` блоке, он победит централизованный конфиг.

На карточке, привязанной к доске, внизу появляется маленькая кнопка «⚙ поля карточки» — открывает ту же модалку настроек доски сразу на секции «Карточка».

Если тег заметки совпадает с тегом какой-нибудь ` ```board ` доски (не `flat`), над полями дополнительно рисуется ряд чипов со всеми колонками этой доски — активная подсвечена, клик по другой сразу переключает `statusField` в frontmatter заметки. Список колонок берётся из `columns` доски, а если он не задан явно — из фактически встречающихся значений `statusField` у её карточек, как и на самой доске.

### Блок `tags`

````markdown
```tags
```
````

Не принимает никакой конфигурации. Вставь в любую заметку — плагин сам найдёт по всему vault ` ```board ` блок, чей `tag` совпадает с одним из тегов текущей заметки, и отрисует редактируемую панель чипов для словарных (`vocab`) полей этой доски — включая небольшую строку с указанием, к какой доске/тегу он подключился, чтобы можно было визуально убедиться в правильности связи. Если подходящая доска не найдена — покажет ошибку, а не тихо ничего не сделает.

Тот же редактор можно вызвать из любой заметки через палитру команд (**Board Notes: Редактировать теги/жанры по словарю доски**) или через правый клик → **Теги/жанры по словарю** в меню файла — пригодится, если в теле заметки нет блока `tags`.

### Настройки доски (⚙)

В тулбаре любой доски (кроме `flat`) есть кнопка ⚙ — открывает модалку настроек прямо поверх code-блока, без ручной правки YAML:

- **Папка** и **Шаблон** — те же значения, что `folder`/`template` в конфиге; рядом с шаблоном — кнопка «+ создать заметку», создающая карточку прямо из модалки.
- **Колонки** — список полей ввода, по одному на колонку:
  - правишь текст в поле → колонка **переименовывается**, и `statusField` обновляется у всех карточек, у которых было старое значение;
  - крестик удаляет колонку — если у неё были карточки, они переезжают в первую из оставшихся колонок (не пропадают из доски);
  - «+ добавить» добавляет пустую колонку в конец списка.
- **Теги / словарь** — то же самое, но для каждого поля из `vocab`: переименование значения массово применяется ко всем карточкам, у которых оно было. Внизу можно добавить совсем новое поле словаря.
- **Карточка** — редактируемые списки для централизованного ` ```card ` (см. выше): «Поля» (простой список), «Ссылки» и «Подписи» (пары поле → подпись). Здесь эти значения не привязаны к конкретным карточкам, поэтому переименование ничего не обновляет по заметкам — просто меняет, что показывает пустой ` ```card ` блок.
- «Сохранить» переписывает сам ` ```board ` code-блок (через `stringifyYaml`) и применяет все переименования к карточкам одной операцией; доска сразу перерисовывается с новым конфигом, без переоткрытия заметки.

Изменить сам `tag` доски или порядок колонок перетаскиванием через эту модалку нельзя — см. «Известные ограничения».

### Создание новой доски

Команда **Board Notes: Создать новую доску** (палитра команд, либо кнопка «+ создать новую доску» внизу модалки настроек) открывает мастер: название заметки, тег, папка для карточек, шаблон (необязательно), колонки (по одной на строку). По кнопке «Создать» генерируется новая заметка с готовым ` ```board ` code-блоком и сразу открывается.

## Как хранятся данные

- **Колонка** = значение поля `statusField` в frontmatter заметки (обычная строка).
- **Позиция внутри колонки** = значение поля `orderField` в frontmatter (целое число, перезаписывается для всей колонки при каждом перетаскивании).
- **Значения, управляемые словарём** = обычные списковые поля frontmatter (или скалярные, для полей из `single`) — сам список словаря хранится только в конфиге блока `board`, не дублируется по заметкам.

Ничего не пишется вне frontmatter самих заметок. Удаление плагина оставляет заметки полностью целыми и читаемыми как обычный YAML frontmatter.

## Известные ограничения

- Одна доска = один code-блок = один тег. Доски, которым нужно сочетать несколько тегов, не поддерживаются.
- Порядок при перетаскивании сохраняется перезаписью поля `orderField` у всех карточек колонки-назначения при каждом drop — нормально для досок с десятками карточек, потенциально медленно при многих сотнях.
- Специального тестирования touch drag-and-drop на мобильных не проводилось.
- Имена полей в `vocab`/`facets` сопоставляются по точному совпадению строки — переименование поля (ключа, не значения) во frontmatter требует правки конфига доски вручную.
- Модалка настроек (⚙) умеет переименовывать *значения* колонок и словаря (с массовым обновлением карточек), но не сам `tag` доски и не имена полей (`statusField`, ключи `vocab`) — это по-прежнему правится вручную в code-блоке.
- Порядок колонок в модалке настроек нельзя менять перетаскиванием — только добавлять/удалять/переименовывать; чтобы изменить порядок, поправь список `columns` в code-блоке.

## Вклад в разработку

Issues и PR приветствуются. Весь код — один файл `main.ts`, без фреймворков поверх esbuild, без сторонних UI-библиотек, только Obsidian API и обычные вызовы DOM.
